[SEGMEN 2 — KONSEP DASAR REQUEST-RESPONSE] (durasi: ~90 detik)
------------------------------------------------------------------

Model pertama yang kita bahas adalah Request-Response.

Ini adalah model komunikasi yang paling banyak kita temui sehari-hari.
Setiap kali Anda membuka browser dan mengakses sebuah website,
atau menggunakan aplikasi mobile yang mengambil data dari internet,
model ini yang bekerja di baliknya.

Konsep dasarnya sangat sederhana:

  Pertama, ada CLIENT. Client adalah pihak yang memulai komunikasi.
  Dalam simulasi ini, client adalah browser React yang berjalan di
  port 5173.

  Kedua, ada SERVER. Server adalah pihak yang menerima permintaan.
  Dalam simulasi ini, server adalah Gin framework Golang yang
  berjalan di port 8080.

  Ketiga, ada DATABASE. Ini adalah penyimpanan data permanen yang
  diakses oleh server saat memproses request. Di sini kita menggunakan
  MySQL database dengan nama 'sister2'.

DIAGRAM ALIR REQUEST-RESPONSE:
[ REACT UI ]                            [ GIN SERVER ]                         [ MYSQL DB ]
  (Client)                                (Backend)                             (Storage)
     |                                        |                                     |
     | 1. HTTP POST /api/request              |                                     |
     |--------------------------------------->|                                     |
     | (MENUNGGU / BLOCKING)                  | 2. Parsing & Validasi JSON          |
     |                                        |                                     |
     |                                        | 3. INSERT (status: "pending")       |
     |                                        |------------------------------------>|
     |                                        |                                     |
     |                                        | 4. Pemrosesan (Delay & Error Rate)  |
     |                                        |                                     |
     |                                        | 5. UPDATE (status: "success/error") |
     |                                        |------------------------------------>|
     | 6. HTTP Response 200 OK / 500 ERR      |                                     |
     |<---------------------------------------|                                     |
     |                                        |                                     |
 (Selesai)                                (Selesai)                             (Selesai)

PENJELASAN DIAGRAM:
  Langkah 1 [Inisiasi]: Client (yakni React UI) menyusun data request berbentuk JSON yang isinya mencakup jalur skenario, payload, serta parameter buatan (error rate dan latensi simulasi). Data ini ditembak menuju endpoint `/api/request` pada server Gin melalui koneksi HTTP POST.

[SEGMEN 3 — DEMO SIMULASI REQUEST-RESPONSE] (durasi: ~120 detik)
------------------------------------------------------------------

[AKSI: Buka browser ke http://localhost:5173]

Ini adalah tampilan simulasi Request-Response yang saya buat.

Anda bisa melihat tiga kolom utama:
  - Kolom kiri adalah panel kontrol untuk mengirim request
  - Kolom tengah adalah diagram animasi dan log real-time
  - Kolom kanan adalah tampilan response dan perbandingan model

Di header, Anda bisa melihat status server Gin yang aktif di port 8080,
dan database MySQL yang terhubung ke sister2.

[AKSI: Pilih skenario "GET /products"]

Pertama, mari kita kirim request GET untuk mengambil daftar produk.
Saya pilih skenario ini dari dropdown.

Perhatikan slider latency. Saya set ke 600 milidetik untuk
mensimulasikan waktu server memproses data.

Error rate saya set ke 0, artinya tidak ada simulasi error.

[AKSI: Klik "Kirim Request"]

Amati diagram di tengah layar.

Pertama, paket data berwarna biru bergerak dari node CLIENT
ke node GIN SERVER. Ini adalah HTTP GET request yang dikirim.

Kemudian warna berubah ke ungu, menandakan server sedang memproses.

Selanjutnya paket bergerak ke kanan, menuju node MYSQL DATABASE.
Server sedang menyimpan data request ke tabel sim_requests.

Setelah itu terjadi proses balik: data dibaca dari database,
dan akhirnya paket berwarna hijau bergerak kembali dari server ke client.

Ini adalah response yang berhasil!

[AKSI: Klik tab Response]

Di panel Response, kita bisa melihat detail lengkap:
  - HTTP Status Code: 200 OK
  - Processing Time yang aktual
  - Request ID yang tersimpan di database
  - Body response berisi daftar produk e-commerce
  - Dan tujuh flow steps yang menggambarkan alur yang baru terjadi

[AKSI: Coba POST /order dengan error rate 20%]

Sekarang mari kita coba POST request dengan error rate 20 persen.
Ini mensimulasikan kondisi server yang tidak sempurna di dunia nyata.

[AKSI: Klik "Kirim Request" dan amati jika error terjadi]

Kalau terjadi error, Anda akan melihat status code 500 atau 503,
yang berarti server mengalami masalah internal.

Inilah salah satu karakteristik Request-Response: jika server mengalami
masalah, client LANGSUNG MENGETAHUINYA karena tidak mendapat response
yang diharapkan.

[SEGMEN 4 — KONSEP DASAR PUBLISH-SUBSCRIBE] (durasi: ~90 detik)
------------------------------------------------------------------

Model kedua adalah Publish-Subscribe, atau biasa disingkat Pub-Sub.

Model ini bekerja dengan cara yang sangat berbeda dari Request-Response.

Bayangkan Anda subscribe ke sebuah channel YouTube. Channel tersebut
tidak tahu persis siapa subscriber-nya. Ketika channel upload video baru,
VIDEO itu otomatis diterima oleh SEMUA subscriber — tanpa channel harus
tahu siapa dan di mana mereka berada.

Itulah inti dari Publish-Subscribe.

DIAGRAM ALIR PUBLISH-SUBSCRIBE:
[ PUBLISHER ]                      [ RABBITMQ ]                              [ CONSUMER SERVICES ]
 (Pemicu)                       (Message Broker)                           (Mendengarkan 24/7)
     |                                  |                                             |
     | 1. Kirim "order.created"         |                                             |
     |--------------------------------->| [ Exchange: Fanout ]                        |
     | (Publisher Tugas Selesai ->      |          |                                  |
     |  langsung lanjut memori lepas)   |          |-- Meneruskan --> Queue: Inventory| -> (Potong Stok)
                                        |          |                                  |
                                        |          |-- Meneruskan --> Queue: Payment  | -> (Cek Invoice Bank)
                                        |          |                                  |
                                        |          |-- Meneruskan --> Queue: Shipping | -> (Siapkan Resi kurir)

PENJELASAN DIAGRAM:
  Langkah 1 [Pendaftaran Broker / Binding]: Keempat *Consumer Service* ini (Inventory, Payment, Shipping, Notification) berjalan LEBIH DAHULU mendahului Publisher. Masing-masing membuka pipa AMQP ke daemon RabbitMQ, mengikat (*binding*) Queue unik individunya ke sebuah pos "Exchange" RabbitMQ bertipe `Fanout` yang bernama `ecommerce_events`. Pasca mendaftar, mereka membungkam diri secara pasif menunggu di thread-nya.

Komponennya terdiri dari:

  PUBLISHER: pihak yang menerbitkan atau mengirim pesan.
  Dalam simulasi saya, publisher adalah publisher.go yang mengirim
  event-event e-commerce ke RabbitMQ.

  MESSAGE BROKER: perantara yang meneruskan pesan dari publisher
  ke semua subscriber. Di sini kita menggunakan RabbitMQ dengan
  tipe exchange FANOUT, artinya pesan disiarkan ke SEMUA queue
  yang terhubung.

  CONSUMER (SUBSCRIBER): pihak yang menerima dan memproses pesan.
  Dalam simulasi saya, ada empat consumer service yang berjalan
  secara paralel dalam goroutine masing-masing:
    - Inventory Service: mengelola stok produk
    - Payment Service: mencatat transaksi keuangan
    - Shipping Service: menyiapkan pengiriman
    - Notification Service: mengirim notifikasi ke pengguna

Yang paling penting: Publisher tidak menunggu Consumer selesai memproses.
Publisher mengirim pesan dan langsung melanjutkan tugasnya.
Ini adalah komunikasi ASINKRON.

[SEGMEN 5 — DEMO SIMULASI PUBLISH-SUBSCRIBE] (durasi: ~120 detik)
------------------------------------------------------------------

[AKSI: Buka terminal, masuk ke direktori publish-subscribe]

Berbeda dengan Request-Response yang memiliki UI visual,
simulasi Publish-Subscribe ini berjalan di terminal karena
fokusnya adalah pada alur pesan antar service.

[AKSI: Jalankan: go run consumers.go --semua]

Terminal pertama menjalankan semua consumer service sekaligus.
Anda bisa melihat keempat service mulai terhubung ke RabbitMQ:
  - Inventory Service menunggu di queue inventory.queue
  - Payment Service menunggu di queue payment.queue
  - Shipping Service menunggu di queue shipping.queue
  - Notification Service menunggu di queue notification.queue

Semua service ini berjalan secara PARALEL menggunakan goroutine Golang.

[AKSI: Buka terminal kedua, jalankan: go run publisher.go]

Terminal kedua menjalankan publisher yang akan mengirim 10 event
secara otomatis.

[AKSI: Amati output di kedua terminal]

Perhatikan apa yang terjadi saat publisher mengirim event "order.created":

Di terminal publisher:
  Dikirim: event=order.created, order=ORD-xxx, produk=Laptop Gaming ASUS

Di terminal consumer, TIGA service langsung merespons secara bersamaan:
  Inventory Service: Stok dikurangi sekian unit
  Shipping Service: Alamat pengiriman disiapkan
  Notification Service: Notifikasi pesanan diterima dikirim ke customer

Perhatikan bahwa Payment Service TIDAK merespons event order.created,
karena Payment Service hanya subscribe ke event payment.success
dan order.cancelled.

Inilah keunggulan Pub-Sub: setiap consumer hanya menerima event
yang relevan dengannya, tanpa publisher perlu tahu siapa yang
membutuhkan event tersebut.

[SEGMEN 6 — PERBEDAAN MENDALAM KEDUA MODEL] (durasi: ~120 detik)
------------------------------------------------------------------

Sekarang mari kita bandingkan kedua model secara mendalam.

ASPEK PERTAMA: Pola Komunikasi

Request-Response bersifat SINKRON.
Client mengirim request dan memblokir eksekusinya, tidak melakukan
apapun, sampai response diterima dari server.
Analogi: Anda menelepon seseorang dan menunggu mereka mengangkat telepon.
Selama menunggu, Anda tidak bisa melakukan hal lain.

Publish-Subscribe bersifat ASINKRON.
Publisher mengirim pesan dan LANGSUNG melanjutkan tugasnya tanpa
menunggu consumer selesai memproses.
Analogi: Anda mengirim email. Setelah terkirim, Anda bisa langsung
melakukan hal lain tanpa menunggu penerima membaca emailnya.

ASPEK KEDUA: Coupling atau Ketergantungan

Request-Response memiliki TIGHT COUPLING (ketergantungan ketat).
Client harus tahu alamat IP, port, dan endpoint server secara exact.
Jika server berganti alamat, client harus diperbarui.

Publish-Subscribe memiliki LOOSE COUPLING (ketergantungan longgar).
Publisher hanya tahu nama exchange di RabbitMQ, tidak tahu siapa
consumer-nya. Anda bisa tambah consumer baru tanpa mengubah kode
publisher sama sekali.

ASPEK KETIGA: Skalabilitas

Karena setiap request membutuhkan satu koneksi HTTP langsung,
Request-Response lebih sulit di-scale untuk volume tinggi.
Server harus menangani setiap request secara real-time.

Karena pesan di-buffer di message broker, Publish-Subscribe jauh
lebih mudah di-scale. RabbitMQ bisa menyimpan ribuan pesan
ketika consumer sedang sibuk, dan consumer bisa dijalankan
di banyak server sekaligus.

ASPEK KEEMPAT: Fault Tolerance atau Toleransi Kegagalan

Ini adalah perbedaan yang sangat krusial di dunia nyata.

Jika server Request-Response mati, semua client langsung merasakan
dampaknya. Request gagal dengan error 503 Service Unavailable.
Tidak ada fallback, tidak ada retry otomatis.

Jika consumer Publish-Subscribe mati, pesan tetap aman tersimpan
di queue RabbitMQ. Saat consumer kembali online, ia akan memproses
semua pesan yang tertunda secara otomatis.
Ini sangat krusial untuk sistem yang membutuhkan keandalan tinggi.

ASPEK KELIMA: Kapan Menggunakan Masing-Masing

Gunakan Request-Response ketika:
  - Anda membutuhkan hasil langsung dari operasi
  - Operasi bersifat transaksional (pembayaran, login, cek stok)
  - Sistem relatif sederhana dengan sedikit komponen
  - Debugging dan monitoring mudah diprioritaskan

Gunakan Publish-Subscribe ketika:
  - Satu event perlu ditangani banyak service berbeda
  - Sistem microservice yang terdistribusi di banyak server
  - Volume pesan sangat besar dan perlu di-buffer
  - Toleransi kegagalan menjadi prioritas utama

[SEGMEN 7 — RELEVANSI DUNIA NYATA] (durasi: ~90 detik)
------------------------------------------------------------------

Kedua model ini bukan hanya teori. Keduanya digunakan secara luas
di perusahaan teknologi besar di seluruh dunia.

RELEVANSI REQUEST-RESPONSE DI DUNIA NYATA:

Setiap kali Anda meminta halaman web dari browser, itu adalah
Request-Response. HTTP sendiri adalah implementasi paling
populer dari model ini.

api.tokopedia.com ketika Anda klik tombol "Beli Sekarang",
aplikasi mengirim POST request ke server, server memvalidasi,
menyimpan order, dan mengembalikan nomor order ke Anda.
Anda menunggu respons itu sebelum diarahkan ke halaman pembayaran.

Google Maps API, Twitter Timeline API, semua RESTful API yang
ada di dunia ini menggunakan Request-Response.

RELEVANSI PUBLISH-SUBSCRIBE DI DUNIA NYATA:

Di Tokopedia, saat order berhasil dibuat:
  - Sistem inventory menerima sinyal untuk mengurangi stok
  - Sistem payment menerima sinyal untuk memproses pembayaran
  - Sistem logistik menerima sinyal untuk menyiapkan pengiriman
  - Sistem notifikasi mengirim email dan push notification ke user

Semua ini terjadi BERSAMAAN, PARALEL, tanpa saling menunggu.
Ini hanya dimungkinkan dengan Publish-Subscribe.

Contoh lain yang lebih familiar:
  - Notifikasi WhatsApp: saat Anda kirim pesan, pesan dipublish,
    dan semua device penerima yang subscribe langsung menerima
  - Feed Instagram: saat akun yang Anda follow posting, semua
    follower menerima update secara bersamaan

[SEGMEN 8 — KEUNGGULAN DAN KELEMAHAN] (durasi: ~60 detik)
------------------------------------------------------------------

REQUEST-RESPONSE
  Keunggulan:
  + Simple dan mudah di-debug (request-response satu arah)
  + Hasil langsung tersedia setelah request
  + Tidak memerlukan infrastruktur tambahan (tidak perlu broker)
  + Mudah dipahami dan diimplementasikan
  
  Kelemahan:
  - Client harus menunggu (blocking) — tidak efisien untuk operasi lambat
  - Jika server down, seluruh alur terhenti
  - Sulit di-scale untuk volume request yang sangat tinggi
  - Tight coupling membuat sistem kurang fleksibel

PUBLISH-SUBSCRIBE
  Keunggulan:
  + Scalable — consumer bisa di-scale horizontal secara independen
  + Fault tolerant — pesan tersimpan di queue meski consumer down
  + Loose coupling — mudah tambah/hapus consumer tanpa ubah publisher
  + Pararel — banyak consumer proses pesan secara bersamaan
  
  Kelemahan:
  - Kompleks — memerlukan infrastruktur message broker
  - Debugging lebih sulit — alur pesan tidak linear
  - Latency lebih tinggi akibat overhead broker
  - Sulit untuk operasi yang membutuhkan konfirmasi langsung


------------------------------------------------------------------
[SEGMEN 9 — PENUTUP] (durasi: ~30 detik)
------------------------------------------------------------------

Jadi, kesimpulannya:

Request-Response dan Publish-Subscribe bukan model yang saling
menggantikan, melainkan saling melengkapi.

Sistem yang baik seringkali menggunakan keduanya sekaligus.
Misalnya, user mengirim order melalui HTTP API (Request-Response),
dan setelah order tersimpan, internal service notification, inventory,
dan shipping dikoordinasikan melalui message broker (Publish-Subscribe).

Itulah yang membuat arsitektur sistem terdistribusi modern menjadi
kuat: memilih model komunikasi yang tepat untuk kebutuhan yang tepat.
